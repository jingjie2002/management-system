## 5. 你现在可以怎么测试

你不是原作者，所以这里按“直接照着点就能测”的方式写。

### 5.1 启动方式

优先使用根目录脚本：

```powershell
start_all_mysql.bat
```

说明：

1. 如果本机 `3306` 上的 MySQL 可用，脚本会按 MySQL 模式启动
2. 如果 MySQL 不可用，脚本会自动切到 H2 模式
3. 后端健康检查脚本当前按 `8080` 检查
4. 前端健康检查脚本当前按 `5173` 检查

如果脚本窗口提示端口或地址与文字说明不完全一致，以脚本实际打开的浏览器地址和健康检查结果为准。

## 6. 当前仍然存在的问题

本轮虽然完成了第一阶段前端闭环，但还有以下问题没有彻底收尾：

### 6.1 前端没有正式自动化测试体系

当前前端主要通过 `npm run build` 做构建级验证，还没有接入 Vitest 或 Playwright 这类正式测试体系。

影响：

1. 当前能证明“页面可编译、接口已接线”
2. 但还不能像后端那样形成前端自动回归测试

### 6.2 前端包体仍然偏大

`npm run build` 仍提示部分 chunk 超过 `500 kB`。

影响：

1. 不影响答辩演示
2. 但不利于论文中“系统性能与工程质量”的表达

### 6.3 启动脚本文案仍有不一致

当前脚本和说明中存在端口文案不完全统一的问题，例如：

1. `start_all.bat` 输出文本里有旧端口信息
2. 健康检查脚本按 `8080 / 5173` 检查

影响：

1. 不一定影响运行
2. 但会影响非原作者使用时的理解

### 6.4 还缺少一次完整人工端到端走查

虽然本轮已经完成构建和后端测试回归，但还没有在浏览器里完成一轮“员工申请 -> HR审批 -> 管理员执行 -> 个人中心联动”的完整人工走查截图留证。
