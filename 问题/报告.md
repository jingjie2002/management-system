# 企业员工精细化人事管理系统长审计报告

审计时间：2026-03-31  
审计对象：`management-system` 项目目录  
审计依据：`基于SpringBoot的企业员工精细化人事管理系统设计与实现-下载评阅稿.docx`、项目源码、配置文件、数据库初始化脚本、自动化验证结果

## 1. 审计范围与方法

本次审计以论文评阅稿中描述的目标系统为基准，对当前仓库中的前端、后端、数据库、权限控制、统计与算法实现、工程化交付情况进行了交叉核查，重点关注以下内容：

1. 论文中承诺的功能模块是否已经在代码中实现。
2. 论文中描述的技术架构、权限模型、算法逻辑是否与代码一致。
3. 项目当前是否处于“可答辩、可演示、可继续开发”中的哪一种状态。
4. 是否存在影响演示、验收、答辩或后续上线的关键风险。

本次审计过程中实际执行了以下验证：

1. 前端执行 `npm run build`，结果为构建成功，但存在体积告警。
2. 后端执行 `./mvnw test`，结果为失败，共 4 个测试、3 个失败。
3. 结合源码与日志，对权限、登录、统计、个人中心、培训、项目、绩效等模块进行逐项核对。

## 2. 总体结论

当前项目已经具备“毕业设计演示原型”的基本雏形，前后端骨架、主要页面、数据库初始化、部分管理流程已经打通，能够支撑有限场景下的功能展示；但如果以论文评阅稿中描述的“企业员工精细化人事管理系统完整实现”作为标准，当前代码库仍然存在明显差距。

综合判断如下：

1. 项目不是空壳，属于“已完成较多页面和接口、但离完整交付仍有明显差距”的中后期版本。
2. 项目可以支撑部分演示，但不宜直接宣称“已完整实现论文全部功能并通过验证”。
3. 当前最严重的问题不是页面缺失，而是“后端验证失败、权限边界不牢、多个关键流程带有演示硬编码、论文中的部分算法与业务承诺没有真正落地”。
4. 如果用于正式验收、论文答辩或继续工程化开发，建议先完成一轮系统整改，而不是直接以现状交付。

一句话评价：当前系统更接近“功能较全的毕业设计演示版”，而不是“论文所描述的完整、严谨、可验证的人事管理系统成品”。

## 3. 项目当前情况判断

### 3.1 已具备的基础能力

从代码结构看，项目具备较完整的前后端分离形态：

1. 后端为 Spring Boot 单体服务，包含认证、用户、角色、部门、员工、考勤、请假、绩效、培训、项目、公告、统计与个人中心等模块。
2. 前端为 Vue 3 + Element Plus 管理后台，主要业务页面已经搭建完成，包含登录、仪表盘、员工、部门、考勤、请假、绩效、培训、项目、公告、用户等页面。
3. 数据层提供了 `schema.sql`，并可通过 H2 或 MySQL 运行。
4. 论文中强调的仪表盘、个人中心、组织树、导入导出等能力，在代码中都能找到对应实现痕迹。

### 3.2 当前可运行性

从实际验证结果看：

1. 前端构建通过，说明基础页面和依赖关系整体可编译。
2. 后端测试失败，说明系统当前不满足“自动化验证通过”的最基本要求。
3. 日志中存在配置与实体映射警告，说明后端虽能运行，但存在结构性瑕疵。

因此，项目当前状态更准确的描述应为：

1. 可运行：是。
2. 可部分演示：是。
3. 可证明实现质量稳定：否。
4. 与论文承诺高度一致：否。

## 4. 关键审计发现

## 4.1 P0 级问题：会直接影响验收、答辩可信度或系统安全

### 4.1.1 后端自动化验证当前失败

实际执行 `./mvnw test` 的结果为：4 个测试中 3 个失败。失败集中在登录、仪表盘访问、个人中心访问。

直接原因是：

1. 测试用例只提交了用户名和密码。
2. 登录接口现在要求验证码。
3. 验证码又依赖 Session 存储。
4. 测试并没有先获取验证码并携带会话，因此直接返回“验证码错误”。

证据：

1. `src/test/java/com/ch/managementsystem/ManagementSystemApplicationTests.java` 中登录测试未处理验证码流程。
2. `src/main/java/com/ch/managementsystem/controller/AuthController.java:34-52` 明确通过 `/api/auth/captcha` 生成验证码并写入 `HttpSession`。
3. `src/main/java/com/ch/managementsystem/controller/AuthController.java:52` 从 Session 读取验证码进行校验。

影响：

1. 论文若声称系统测试通过，则与当前仓库状态不一致。
2. 任何自动化验收或 CI 都会被此问题直接拦住。

### 4.1.2 认证设计自相矛盾：声明无状态 JWT，实际又依赖 Session 验证码

系统安全配置声明使用 JWT 无状态认证，但登录阶段又依赖 Session 保存验证码，形成设计冲突。

证据：

1. `src/main/java/com/ch/managementsystem/config/SecurityConfig.java:55` 设置 `SessionCreationPolicy.STATELESS`。
2. `src/main/java/com/ch/managementsystem/controller/AuthController.java:37` 将验证码写入 Session。
3. `frontend/src/api/http.ts:63` 设置 `withCredentials: true`，前端必须携带 Cookie 才能完成验证码登录。

这说明当前认证体系并不是纯 JWT，也不是清晰的 Session 方案，而是两种模式混用。

影响：

1. 自动化测试容易失效。
2. 部署到跨域环境、反向代理或严格无状态架构时容易出问题。
3. 与论文中“JWT + RBAC”这类简洁架构描述不完全一致。

### 4.1.3 后端 RBAC 没有真正落实，多个敏感模块缺少服务端权限保护

论文强调基于角色的权限控制，但当前后端只有部分模块做了 `@PreAuthorize`，多个高风险控制器没有任何角色限制。

高风险点包括：

1. `src/main/java/com/ch/managementsystem/controller/RoleController.java:13` 没有 `@PreAuthorize`。
2. `src/main/java/com/ch/managementsystem/controller/NoticeController.java:18` 没有 `@PreAuthorize`。
3. `src/main/java/com/ch/managementsystem/controller/PerformanceController.java:21` 没有 `@PreAuthorize`。
4. `src/main/java/com/ch/managementsystem/controller/AttendanceController.java:16` 没有 `@PreAuthorize`。
5. `src/main/java/com/ch/managementsystem/controller/LeaveController.java:14` 没有 `@PreAuthorize`。
6. `src/main/java/com/ch/managementsystem/controller/TrainingController.java:15` 没有 `@PreAuthorize`。
7. `src/main/java/com/ch/managementsystem/controller/UserController.java:37`、`src/main/java/com/ch/managementsystem/controller/UserController.java:42` 的 `getUserById` 与 `getUserInfo` 也未见角色限制。

这意味着：

1. 只要用户已经登录，很多本应只有管理员或人事可操作的接口，理论上都可能被普通账号访问。
2. 当前权限更像“前端页面是否显示”的展示性权限，而不是“后端强制拦截”的真实权限。

这是本次审计中最严重的安全问题之一。

### 4.1.4 多个业务流程仍保留演示硬编码，系统行为不依赖真实当前用户

当前前端存在多处明显的演示型硬编码，表明系统并未完全进入真实业务流。

证据：

1. `frontend/src/views/attendance/index.vue:104` 签到时直接提交 `employeeId: 1`。
2. `frontend/src/views/attendance/index.vue:115` 签退直接取表格第一行 `tableData.value[0]?.id`。
3. `frontend/src/views/leave/index.vue:122` 请假表单默认 `employeeId: 1`。
4. `frontend/src/views/leave/index.vue:153` 审批人直接写死为 `'admin'`。
5. `frontend/src/views/Dashboard.vue:150` 通过 `localStorage.setItem('switchTo', username)` 做“角色切换”。
6. `frontend/src/views/Login.vue:82` 直接展示 `employee / 123456` 等演示账号入口。

影响：

1. 当前部分业务动作不是“当前登录用户驱动”，而是“固定测试数据驱动”。
2. 演示时可能看起来能跑，但真实用户使用后数据会错位。
3. 论文若强调“精细化管理”和“角色隔离”，这些硬编码会削弱可信度。

## 4.2 P1 级问题：功能存在，但与论文要求或业务含义明显偏离

### 4.2.1 仪表盘算法实现与论文描述不一致

论文中明确提到：

1. 使用最小堆筛选部门考勤率 Top5。
2. 使用加权贡献值算法计算员工贡献度。
3. 权重为考勤率、绩效、项目贡献等组合。

但代码中的实际实现存在明显偏差。

证据：

1. `src/main/java/com/ch/managementsystem/service/impl/StatisticsServiceImpl.java:118` 月度绩效趋势取的是 `avg(base_salary)`，不是绩效得分。
2. `src/main/java/com/ch/managementsystem/service/impl/StatisticsServiceImpl.java:132` 当数据为空时，直接生成模拟趋势数据。
3. `src/main/java/com/ch/managementsystem/service/impl/StatisticsServiceImpl.java:173` 部门考勤率统计，只是看员工在某时间段内“是否有考勤记录”，并不是严格意义上的考勤率。
4. `src/main/java/com/ch/managementsystem/service/impl/StatisticsServiceImpl.java:199` 员工考勤分数只按“有记录=100，无记录=0”处理。
5. `src/main/java/com/ch/managementsystem/service/impl/StatisticsServiceImpl.java:201` 项目贡献分数实际上按“担任项目经理的项目数 * 20”估算，且上限 100。
6. `src/main/java/com/ch/managementsystem/utils/AlgorithmUtils.java:33` 实际权重为 `0.3 / 0.5 / 0.2`。

这与论文中“更精细的考勤率、绩效值、项目贡献值融合模型”存在偏差，且权重也不一致。

结论：

当前仪表盘更像“有统计展示效果的业务近似实现”，不是论文意义上的严谨算法落地。

### 4.2.2 绩效模块并未真正实现“绩效”而更像“工资/出勤说明”

绩效模块虽然存在，但核心计算逻辑偏弱，和论文所承诺的绩效考核深度不一致。

证据：

1. `src/main/java/com/ch/managementsystem/service/impl/PerformanceServiceImpl.java:42` 存在自动计算入口。
2. `src/main/java/com/ch/managementsystem/service/impl/PerformanceServiceImpl.java:74-76` 只计算出勤奖励、迟到扣款、请假扣款等衍生值。
3. `src/main/java/com/ch/managementsystem/service/impl/PerformanceServiceImpl.java:80` 最终主要是拼接说明字符串。
4. `src/main/java/com/ch/managementsystem/service/impl/PerformanceServiceImpl.java` 中虽然实体包含 `projectBonus`，但核心逻辑并未真正计算并写入项目贡献奖金。
5. `frontend/src/views/performance/PerformanceCalculator.vue:6`、`:25` 仍引用 `projectBonus` 与 `otherDeductions`，但该组件并未被实际业务页面使用。

问题本质：

1. 当前模块更像“按工资和考勤生成月度记录”。
2. 缺少论文常见的人事绩效要素，如指标评分、等级评定、绩效面谈、项目参与权重、人工复核流程等。
3. 前后端字段并未完全对齐。

### 4.2.3 培训模块只实现了基础 CRUD，未达到论文描述的流程深度

论文中除培训档案外，还涉及报名审核、签到管理、培训评价等更完整流程；当前代码仅体现培训记录的增删改查。

证据：

1. `src/main/java/com/ch/managementsystem/controller/TrainingController.java:15` 为单纯培训管理控制器。
2. 前后端都未看到培训报名、审批、签到、评价、学时累计等配套实现。

结论：

培训模块目前属于“页面和表结构意义上的存在”，但业务闭环并未形成。

### 4.2.4 项目模块缺少任务跟踪与参与明细维护，个人中心的项目贡献缺少真实业务来源

论文中提到项目管理、任务跟踪、项目状态等内容，但当前实现仍然偏轻。

问题表现：

1. 项目模块主要是项目基本信息 CRUD。
2. 未看到项目任务分解、责任人进度、里程碑跟踪等能力。
3. 个人中心虽然展示了项目贡献，但前端未见项目参与关系维护入口。

这会导致个人中心中的项目贡献值更多依赖初始化数据，而不是系统持续运营中形成的数据。

### 4.2.5 用户与角色管理模块不完整

论文要求的后台用户管理与角色分配，在后端接口上部分存在，但前端管理能力不完整。

证据：

1. `frontend/src/views/user/UserManagement.vue` 当前主要是列表刷新和启停状态切换。
2. 未形成完整的用户新增、编辑、删除、角色分配、角色管理闭环界面。
3. 与后端 `RoleController`、用户角色分配能力之间存在明显前后端落差。

结论：

用户与角色模块当前只能算“半成品”，离论文中完整的权限管理后台还有差距。

### 4.2.6 部门与员工模块虽较完整，但“精细化人事管理”仍不足

当前部门、员工模块是项目里完成度较高的部分，但仍存在和论文定位不一致的地方。

表现为：

1. 员工实体字段不少，但前端主表单主要暴露姓名、部门、工号、岗位、工资、状态等基础字段。
2. 入职日期、离职日期、身份证、住址、更多档案字段没有形成完整前端维护闭环。
3. 部门负责人目前更像文本录入，而不是和员工档案建立强关联。
4. 论文中提到的部门统计、批量排序调整等更细能力，现有界面并未完整体现。

因此，这部分虽然“可用”，但离“精细化”仍有距离。

## 4.3 P1 级问题：数据一致性与业务可信度风险

### 4.3.1 个人中心存在按姓名自动绑定员工的风险逻辑

`EmployeeServiceImpl` 中如果用户没有绑定员工记录，会尝试按 `sysUser.realName == employee.name` 去匹配员工，再回写 `userId`。

这类逻辑的问题是：

1. 同名员工会误绑定。
2. 演示数据中默认员工账号可能被绑定到不合理的人事记录。
3. 真实场景下会造成身份与档案错配。

关联证据：

1. `src/main/java/com/ch/managementsystem/service/impl/EmployeeServiceImpl.java` 中存在按姓名匹配并写回 `userId` 的逻辑。
2. `src/main/java/com/ch/managementsystem/config/DemoAccountInitializer.java` 中演示员工账号实名为 `John Doe`。
3. `src/main/resources/schema.sql` 中名为 `John Doe` 的员工记录状态为离职。
4. 历史日志 `backend_for_thesis.log:3349` 中也能看到个人中心返回 `employeeId=2, name=John Doe` 的记录。

这意味着演示账号和实际员工档案之间存在可疑绑定，不利于论文答辩中的数据说服力。

## 4.4 P2 级问题：工程化与配置质量不足

### 4.4.1 配置文件存在重复、默认敏感值和混乱依赖

证据：

1. `src/main/resources/application.properties:12` 与 `:69` 都出现了 `spring.autoconfigure.exclude`，后者会覆盖前者，存在配置歧义。
2. `src/main/resources/application.properties:9` 开启了 `spring.main.allow-bean-definition-overriding=true`，说明配置层面存在潜在冲突容忍。
3. `src/main/resources/application.properties:54` 直接写了演示型 JWT 密钥。
4. `src/main/resources/application-mysql.properties:4` 将默认数据库密码写为 `123456`。
5. `src/main/resources/application-dev.properties:11` 设置 `spring.sql.init.continue-on-error=true`，可能掩盖初始化脚本问题。

说明：

1. 当前配置更偏本地演示便利性。
2. 不适合作为严肃交付版本的配置基础。

### 4.4.2 依赖组合不够收敛，存在“能跑但不干净”的迹象

证据：

1. `pom.xml:39` 引入了 `spring-boot-starter-data-jdbc`。
2. `pom.xml:55` 引入 `mybatis-plus-boot-starter`。
3. `pom.xml:60` 又引入 `mybatis-spring-boot-starter`。
4. `pom.xml:134-135` 还引入了 `spring-ai-bom`，但主配置里又排除了多项 Spring AI 自动配置。

这反映出依赖层面存在“叠加式试验”痕迹，后续维护成本会偏高。

### 4.4.3 启动日志已经暴露实体映射和配置告警

证据：

1. `backend_startup.log:30` 和 `backend_for_thesis.log:96` 均提示 `SysUserRole` 找不到主键。
2. 启动日志还出现过 mapper 资源匹配相关警告。

说明数据库映射模型并不完全整洁，虽然未必阻塞运行，但属于明显技术债。

## 4.5 P3 级问题：仓库卫生、测试覆盖与可维护性一般

### 4.5.1 自动化测试覆盖明显不足

当前 `src/test` 下仅看到一份测试文件，且主要覆盖的是登录、仪表盘、个人中心的基础接口联通。

问题在于：

1. 模块很多，但测试很少。
2. 权限、导入导出、培训、项目、请假审批、绩效重算等核心流程没有形成回归测试。
3. 一旦改动登录或权限逻辑，就很容易像这次一样把测试打坏。

### 4.5.2 仓库中混入较多生成物和运行痕迹

从当前仓库情况看，`target`、前端构建产物、`data` 数据文件、日志等内容管理不够严格，`.gitignore` 也没有把前端 `node_modules`、`frontend/dist`、数据目录和日志目录完整排除。

这会带来：

1. 仓库体积膨胀。
2. 提交污染。
3. 审计和交接时难以区分源码与运行产物。

### 4.5.3 源码中保留了调试式输出和异常直打

证据：

1. `src/main/java/com/ch/managementsystem/service/impl/StatisticsServiceImpl.java` 中保留多处 `System.out.println`。
2. `src/main/java/com/ch/managementsystem/service/impl/EmployeeServiceImpl.java:333`、`:371` 以及 `src/main/java/com/ch/managementsystem/utils/CaptchaUtils.java:86` 使用 `printStackTrace()`。

说明代码还没有完全从调试阶段收口到正式交付质量。

### 4.5.4 前端构建虽通过，但包体偏大

`npm run build` 虽然成功，但构建结果已经给出 chunk 超过 500 kB 的警告，主包体积已经偏大。

这不一定影响答辩演示，但说明后续如果继续扩展，前端性能与拆包策略仍需要优化。

## 7. 整改优先级建议

建议按以下顺序整改，而不是平均用力：

### 第一优先级：先修可信度问题

1. 修复登录测试与验证码流程冲突，让 `./mvnw test` 重新通过。
2. 统一认证方案，决定是纯 JWT 还是 Session + JWT，不要继续混用。
3. 为角色、公告、绩效、考勤、请假、培训、用户详情等接口补齐 `@PreAuthorize`。
4. 清理前端中 `employeeId: 1`、`approver: 'admin'`、假角色切换等演示硬编码。

### 第二优先级：补齐论文最容易被追问的模块

1. 重新梳理仪表盘统计口径，避免用工资均值替代绩效值。
2. 去掉或显式标注模拟数据逻辑，避免答辩时被质疑统计真实性。
3. 补齐培训报名、审核、签到、评价等流程，或者在论文中收缩表述。
4. 补齐项目任务/参与维护入口，打通个人中心项目贡献的数据来源。

### 第三优先级：提升工程质量

1. 收敛依赖，清理不必要 starter 与 AI 相关残留。
2. 整理配置文件，移除重复项、弱口令默认值和不必要的覆盖开关。
3. 扩充自动化测试，至少覆盖权限、审批、统计、导入导出等高风险流程。
4. 清理仓库中的构建产物、日志与数据文件，完善 `.gitignore`。
5. 将 `System.out.println` 与 `printStackTrace()` 改为统一日志方案。

## 8. 最终结论

当前项目已经具备一定完成度，能够说明作者确实进行了较多开发工作，绝非简单模板拼接；但如果严格依据论文评阅稿来判断，“系统已完整实现并稳定通过验证”的结论目前站不住。

最核心的判断是：

1. 项目有基础，有内容，有一定业务覆盖。
2. 项目尚未彻底完成论文承诺的全部深度能力。
3. 项目最需要补的不是“再加几个页面”，而是“修正安全边界、打通真实业务流、校正统计算法、恢复测试可信度”。

因此，建议将该项目当前状态定性为：

“一个功能较丰富、可演示、但尚未完成系统化收口与论文一致性修正的毕业设计项目。”
