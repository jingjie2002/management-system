# 第2次报告

时间：2026-03-31  
主题：围绕导师评语补齐“转”流程，并补充非原作者可执行的测试说明

## 1. 本次工作的目标

根据导师评语，当前项目需要从“有若干管理页面”继续提升到“覆盖入、转、离、考、绩全流程的人事系统”。

本次先聚焦导师明确点名的“转”：

1. 岗位调动
2. 职级调整
3. 部门调动
4. 薪资联动
5. 异动记录留痕

本次阶段性目标不是把全系统一次性重做，而是先把“转”做成一个可以答辩、可以演示、可以写进论文的完整闭环。

## 2. 当前结论

结合前次审计和本轮沟通，当前项目在“转”这一块的真实状态是：

1. 代码中只有员工主档字段，如 `deptId`、`position`、`baseSalary`，没有独立的异动流程模型。
2. 系统没有“员工申请 -> 审批 -> 执行生效 -> 留痕”的完整异动链路。
3. 没有职级主数据管理能力，无法支撑职级排序、职级调整和与薪资区间联动。
4. 目前的“转”更像直接修改员工数据，而不是一个正式的人事业务流程。

因此，如果不补这一块，项目与导师要求以及论文层次之间会一直存在明显缺口。

## 3. 本阶段确认后的整改方向

本轮已经和你确认，第一阶段采用“独立异动中心”方案，主要设计如下：

1. 新增“职级管理”模块
2. 系统内置可演示的职级数据，但这些数据允许管理员/HR新增、修改、删除、停用
3. 新增“员工异动管理”模块，承载转岗、调级、调部门、调薪
4. 同时支持“员工发起申请”和“HR/管理员直接办理”两种入口
5. 只有在“执行生效”节点才回写员工正式档案
6. 全程保留审批意见、执行记录和时间线
7. 权限校验放到后端强制执行，而不只是前端隐藏按钮

## 4. 第一阶段拟达到的效果

这一阶段做完后，系统应当能展示以下闭环：

1. 管理员/HR维护企业的职级体系
2. 员工提交转岗/调级/调部门/调薪申请
3. HR或管理员审批申请
4. HR或管理员执行生效
5. 员工档案自动更新为新的部门、岗位、职级、基础薪资
6. 系统保留完整异动历史，可用于查询、答辩展示和后续统计

## 5. 非原作者可执行的测试说明

你提到这个项目不是你本人开发的，因此这里单独给出“怎么跑、怎么测”的说明。后续功能整改完成后，我也会继续按这个风格补测试清单。

### 5.1 先说结论

当前仓库里虽然有多份一键启动脚本，但脚本之间存在少量不一致信息。为了避免你被误导，现阶段建议优先使用“手动分开启动”的方式测试，这样最稳定、最好排错。

推荐顺序：

1. 先用默认 H2 模式跑通项目
2. 再根据需要切换到 MySQL 模式

### 5.2 默认测试方式：H2 模式

这是当前最省事的方式，不依赖你本机先准备 MySQL 数据库。

#### 启动后端

在项目根目录运行：

```bat
run_backend.bat
```

说明：

1. 该脚本会调用 `mvnw.cmd spring-boot:run`
2. 默认使用 `application.properties` 中的 `dev` 配置
3. 当前默认端口是 `8080`

判断后端启动成功的方式：

1. 终端出现 Spring Boot 启动日志
2. 浏览器可打开 `http://localhost:8080/swagger-ui/index.html`

#### 启动前端

另开一个终端，在项目根目录运行：

```bat
run_frontend.bat
```

说明：

1. 该脚本会进入 `frontend` 目录
2. 自动执行 `npm install`
3. 然后执行 `npm run dev`
4. 当前 Vite 配置端口是 `5173`

判断前端启动成功的方式：

1. 浏览器可打开 `http://localhost:5173`
2. 登录页可以正常显示用户名、密码、验证码输入框

### 5.3 MySQL 模式测试方式

如果你希望用 MySQL 测试，可以使用：

```bat
run_backend_mysql.bat
```

或者：

```bat
start_all_mysql.bat
```

但要先满足下面条件：

1. 本机 3306 端口有 MySQL 服务
2. 数据库名为 `management_system`
3. 默认用户名密码为 `root / 123456`

对应配置位置：

1. `src/main/resources/application-mysql.properties`
2. 数据库连接可通过环境变量 `DB_URL`、`DB_USERNAME`、`DB_PASSWORD` 覆盖

### 5.5 当前系统如何做基础手工测试

如果你只是想确认“项目现在能不能打开、主要页面能不能点”，可以按下面走：

1. 启动后端：`run_backend.bat`
2. 启动前端：`run_frontend.bat`
3. 打开 `http://localhost:5173`
4. 使用 `admin / 123456` 登录，并输入页面验证码
5. 检查仪表盘是否能加载
6. 依次点击部门、员工、考勤、请假、绩效、培训、项目、用户等页面，看是否能正常打开
7. 使用 `hr / 123456` 再登录一次，观察菜单范围变化
8. 使用 `employee / 123456` 再登录一次，观察菜单范围变化

### 5.6 当前仓库测试时要知道的几个坑

1. `./mvnw test` 当前会失败
原因：登录测试没有处理验证码和 Session，导致返回“验证码错误”。

2. `health_check.bat` 的登录检查可能会告警
原因：它直接请求登录接口，没有走验证码流程。

3. `start_all.bat` 中打印的端口信息与当前实际配置不完全一致
脚本里显示了 `38889` 和 `5174`，但当前配置和健康检查主要还是基于 `8080` 和 `5173`。因此更建议你使用 `run_backend.bat` 与 `run_frontend.bat` 这种手动分开启动方式。

## 6. “转”模块完成后的建议测试清单

等第一阶段整改完成后，建议你按下面顺序测试“转”：

### 6.1 管理员/HR测试

1. 登录 `admin` 或 `hr`
2. 进入“职级管理”
3. 新增一个演示职级
4. 编辑一个已有职级
5. 停用或删除一个不再使用的职级
6. 进入“员工异动管理”
7. 直接创建一张异动单，修改部门、岗位、职级和薪资
8. 将异动单审批并执行生效
9. 打开员工档案，确认正式信息已更新
10. 打开异动详情，确认时间线和前后对比完整保留

### 6.2 员工测试

1. 登录 `employee`
2. 进入“我的异动”
3. 提交一张异动申请
4. 确认自己只能看到自己的申请
5. 确认自己看不到职级管理和审批按钮
6. 等 HR 审批后，再查看状态是否变化

### 6.3 权限测试

1. 用员工账号尝试访问审批接口，应被拒绝
2. 用员工账号尝试维护职级，应被拒绝
3. 只有 HR/管理员才能执行生效
4. 未执行前，员工正式档案不能被直接改掉
