# Transfer Center Design

## 背景

导师评语明确指出，系统需要覆盖“入、转、离、考、绩”全流程，并特别点名“转”应体现岗位调动与职级调整。当前项目虽然已经具备员工、部门、考勤、请假、绩效、培训、项目等模块，但“转”仍停留在直接修改员工档案字段的层面，缺少申请、审批、执行、生效和留痕的流程闭环。

本设计文档用于定义第一阶段的“员工异动中心”，将“转”从字段修改提升为正式的人事业务流程，并兼顾答辩展示、论文描述一致性和后续工程扩展性。

## 目标

第一阶段目标：

1. 将岗位调动、职级调整、部门调动、薪资联动统一纳入“员工异动”流程。
2. 同时支持“员工发起申请”和“HR/管理员直接办理”两种业务入口。
3. 新增可维护的职级体系，不写死企业职级规则。
4. 只有在“执行生效”节点才更新员工正式档案。
5. 为每一次异动保留完整时间线、审批意见和前后差异，支撑论文中的流程化与可追溯描述。

不在第一阶段完成的内容：

1. 不把“入、离、转”一次性抽象成统一流程引擎。
2. 不在本阶段完成所有统计图表改造，只保留可扩展接口。
3. 不在本阶段同步重做全部旧模块，只聚焦“转”的完整闭环与权限质量。

## 设计原则

1. 流程优先
“转”必须是业务流程，不再是员工主表字段直接修改。

2. 数据分层
申请数据、审批数据、执行数据与员工正式档案分离，避免流程中间态污染正式数据。

3. 权限双重控制
前端根据角色展示菜单与按钮；后端根据认证身份和角色做强制拦截。

4. 可配置而非写死
职级数据提供演示初始值，但允许管理员和 HR 自主管理。

5. 答辩友好
页面、状态、时间线、前后对比要足够直观，便于非开发者理解系统价值。

## 业务范围

第一阶段“转”覆盖以下业务动作：

1. 部门调动
2. 岗位调动
3. 职级调整
4. 基础薪资联动调整
5. 员工自主申请异动
6. HR 或管理员直接创建异动单
7. HR 或管理员审批异动单
8. HR 或管理员执行异动生效
9. 查询异动历史与时间线

## 核心数据模型

### 1. 职级字典

新增职级字典表，用于维护企业自己的职级体系。系统可初始化一套演示数据，但管理员和 HR 可以新增、修改、删除、停用。

建议字段：

1. `id`
2. `levelCode`
3. `levelName`
4. `levelRank`
5. `salaryMin`
6. `salaryMax`
7. `status`
8. `remark`
9. `createTime`
10. `updateTime`
11. `deleted`

说明：

1. `levelRank` 用于排序和判断职级高低。
2. `salaryMin`、`salaryMax` 用于调级时提示建议薪资区间。
3. 该表属于主数据，不绑定某个单独员工。

### 2. 员工主档新增职级关联

员工表需要新增当前职级关联，例如：

1. `jobLevelId`

这样员工当前职级不再依赖自由文本，而是指向职级字典。岗位 `position` 继续保留，表达具体职位名称；职级表达职位层级。

### 3. 员工异动主表

新增员工异动主表，承载一次完整的“转”流程。

建议字段：

1. `id`
2. `employeeId`
3. `changeType`
4. `applyMode`
5. `status`
6. `reason`
7. `effectiveDate`
8. `beforeDeptId`
9. `afterDeptId`
10. `beforePosition`
11. `afterPosition`
12. `beforeJobLevelId`
13. `afterJobLevelId`
14. `beforeSalary`
15. `afterSalary`
16. `applicantUserId`
17. `approverUserId`
18. `approveRemark`
19. `approveTime`
20. `executeUserId`
21. `executeTime`
22. `createTime`
23. `updateTime`
24. `deleted`

字段含义：

1. `changeType` 允许值建议为：`DEPARTMENT_TRANSFER`、`POSITION_TRANSFER`、`JOB_LEVEL_ADJUSTMENT`、`SALARY_ADJUSTMENT`、`COMPOSITE_CHANGE`。
2. `applyMode` 允许值建议为：`EMPLOYEE_APPLY`、`HR_DIRECT`、`ADMIN_DIRECT`。
3. 一张异动单可以同时修改多个维度；当修改维度大于一个时，`changeType` 使用 `COMPOSITE_CHANGE`。

### 4. 员工异动日志表

新增异动日志表，用于记录操作时间线。

建议字段：

1. `id`
2. `changeId`
3. `actionType`
4. `operatorUserId`
5. `operatorName`
6. `content`
7. `createTime`

日志示例：

1. 员工提交申请
2. HR 审批通过
3. HR 执行生效
4. 管理员驳回申请
5. 员工撤回申请

## 状态设计

统一状态如下：

1. `DRAFT`：草稿
2. `PENDING_APPROVAL`：待审批
3. `REJECTED`：已驳回
4. `APPROVED_PENDING_EFFECTIVE`：已批准待生效
5. `EXECUTED`：已执行
6. `CANCELLED`：已撤回

状态流转规则：

1. 员工申请提交后进入 `PENDING_APPROVAL`
2. HR/管理员直接创建时可直接进入 `APPROVED_PENDING_EFFECTIVE`
3. 审批驳回进入 `REJECTED`
4. 审批通过进入 `APPROVED_PENDING_EFFECTIVE`
5. 执行后进入 `EXECUTED`
6. 申请人在未审批前可撤回，进入 `CANCELLED`

## 流程设计

### 流程 1：员工发起

1. 员工打开“我的异动”
2. 选择异动类型与目标信息
3. 系统自动带出当前部门、岗位、职级、薪资
4. 员工填写申请理由与生效日期
5. 提交后生成异动单，状态为 `PENDING_APPROVAL`
6. 系统写入日志“员工提交申请”

### 流程 2：HR/管理员直接办理

1. HR 或管理员打开“员工异动管理”
2. 选择员工并填写调整信息
3. 系统自动带出调整前信息
4. 可直接保存为 `APPROVED_PENDING_EFFECTIVE`
5. 系统写入日志“HR/管理员直接创建异动单”

### 流程 3：审批

1. HR 或管理员查看待审批异动单
2. 进行通过或驳回
3. 通过后写入审批意见与审批时间，状态改为 `APPROVED_PENDING_EFFECTIVE`
4. 驳回后写入审批意见与审批时间，状态改为 `REJECTED`

### 流程 4：执行生效

1. 只有 HR 或管理员可执行
2. 执行时系统更新员工正式档案：
   - `deptId`
   - `position`
   - `jobLevelId`
   - `baseSalary`
3. 更新成功后，异动单状态改为 `EXECUTED`
4. 写入日志“异动执行生效”

执行前不得修改员工正式档案。这是本设计最关键的边界之一。

## 权限设计

### 员工

允许：

1. 查看自己的异动单
2. 新增自己的异动申请
3. 查看异动详情和时间线
4. 撤回未审批的申请

禁止：

1. 查看全量异动数据
2. 审批异动
3. 执行异动生效
4. 维护职级字典

### HR

允许：

1. 查看全量异动数据
2. 发起异动单
3. 审批异动单
4. 执行异动生效
5. 维护职级字典

### 管理员

拥有 HR 的全部能力，并保留全局管理权限。

### 权限落点

权限必须落在后端控制器与服务层，不得只依赖前端菜单显示。

前端负责：

1. 菜单隔离
2. 按钮显示与隐藏

后端负责：

1. 接口鉴权
2. 越权拦截
3. 关键状态流转校验

## 页面设计

### 1. 职级管理

页面职责：

1. 展示职级列表
2. 新增职级
3. 编辑职级
4. 停用/启用职级
5. 删除职级
6. 展示默认薪资区间

设计要求：

1. 预置一组演示数据
2. 演示数据与新增数据同样可编辑
3. 页面支持按 `levelRank` 排序展示

### 2. 员工异动管理

页面职责：

1. 展示全量异动单
2. 支持按状态、员工、部门、异动类型筛选
3. 支持 HR/管理员直接创建异动单
4. 支持审批、驳回、执行
5. 支持查看详情与时间线

### 3. 我的异动

页面职责：

1. 员工查看自己的异动申请
2. 员工发起申请
3. 员工查看审批进度
4. 员工撤回未审批申请

### 4. 员工档案联动展示

员工详情页或个人中心后续应增加：

1. 当前职级展示
2. 近几次异动记录展示

第一阶段可以先把数据链路打通，页面扩展在第二阶段增强。

## 与现有项目的集成方式

基于当前项目结构，第一阶段将沿用现有 Spring Boot + Vue3 + MyBatis-Plus 模式，不额外引入复杂框架。

后端新增模块：

1. 职级实体、Mapper、Service、Controller
2. 异动实体、Mapper、Service、Controller
3. 异动日志实体、Mapper、Service
4. 员工服务中的“执行异动后回写员工主档”逻辑

前端新增模块：

1. 职级管理页面
2. 员工异动管理页面
3. 我的异动页面
4. 路由与菜单接入
5. 相关类型定义与 API 调用封装

## 测试与验收标准

第一阶段至少需要覆盖以下手工验收路径：

1. 管理员/HR 能新增、修改、删除、停用职级
2. 员工能提交异动申请
3. HR/管理员能审批员工申请
4. HR/管理员能直接创建异动单
5. HR/管理员能执行生效
6. 执行后员工正式档案被正确更新
7. 异动详情能看到前后对比与时间线
8. 员工无法审批或维护职级
9. 未执行前员工正式档案不应提前变化

自动化验证目标：

1. 后端至少补充异动流程核心测试
2. 权限边界至少覆盖员工越权审批、越权维护职级两条
3. 现有登录测试问题后续应单独修复，但不阻碍第一阶段设计成立

## 分阶段实施边界

### 第一阶段

完成：

1. 职级字典
2. 异动主流程
3. 双入口
4. 审批与执行
5. 员工主档联动
6. 时间线
7. 权限补齐

### 第二阶段

再扩展：

1. 异动统计进入仪表盘
2. 个人中心显示异动历史
3. 与绩效、组织、培训等模块更深联动
4. 继续消化前次审计报告中的其它问题

## 风险与约束

1. 当前仓库已有较多历史改动与演示逻辑，实施时需要避免误伤已有功能。
2. 登录验证码与自动化测试之间存在冲突，后续需要单独处理测试适配。
3. 现有前后端权限控制不完全一致，实施时必须优先把异动模块的权限边界做扎实。

## 结论

本设计将“转”从员工字段修改升级为正式的人事异动中心，既满足导师对岗位调动与职级调整的要求，也能与论文中“流程化、权限到人、数据可追踪、降低管理成本”的表述形成一致。第一阶段优先把异动中心做成可演示、可验证、可追溯的闭环，再进入更深层的系统联动改造。
